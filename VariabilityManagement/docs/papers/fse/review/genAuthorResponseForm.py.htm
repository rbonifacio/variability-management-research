          >>> Summary of the submission <<<

This paper presents a variability management framework for Software Product
Lines. In particular, use case scenario variability management is advocated -
from a use case model, feature model, product configuration, and configuration
knowledge the product specific use case model is obtained. The main
contribution of this paper is to show that there is a clear separation between
variability management and scenario specification (although this contribution
was already presented in authors' workshop paper at AOSD'08) and a description
of the framework for modeling the composition process of scenario variability
mechanisms (particularly, how selection of optional scenarios, scenario
composition, and scenario parameterization are performed in this framework). 

After briefly introducing the problem statement, the motivating example (eShop
Product Line) is given. In section 3, the core of the paper is discussed –
first, variability as crosscutting is briefly introduced, and then explained on
a running example. The main part of this section is a description of the
modeling framework by explaining implementation of different weavers (product
derivation weaver, scenario composition weaver, bind parameters weaver). A
description of model verification is also given in this section. In section 4,
evaluation of the proposed approach is given by DSM (Design Structure Matrices)
analysis and quantitative analysis using different measure (e.g., feature
diffusion over use case, number of features pre use case). The paper ends with
a short description of related work and a conclusion. 



          >>> Evaluation <<<

The idea that is explored in this paper is quite novel an unique – the
consideration of variability management at the use case level may offer
benefits, especially with the separation of the variability from the scenario
specification. Although I liked the concepts explored in the paper, I still
have some concerns, as listed below.

I did not find the introduction as appealing as it could be. I think the
authors sort of sell their idea a little short by the manner in which the
introduction is written. The authors do not stress enough (in my opinion) the
importance of the problem they are dealing with. From the current description I
can see that the problem certainly exists and that some other researchers
proposed a solution, too. But, I can not see how crucial this problem is for
real software developers. How important is to solve this problem (I believe it
is very important, but the paper is not as strong in making the case as could
be)? 

The authors have already shown that there is a clear separation between
variability management and scenario specification in their AOSD'08 workshop
paper (ref [7] I this paper). Despite that, it is clear that this paper is a
sufficient extension of the workshop paper. What is new in this paper appears
to be the description of the modeling framework (section 3.2) by showing
implementation of different weavers (product derivation weaver, scenario
composition weaver, bind parameters weaver). The rest of the paper is some
minor improvement over workshop paper.

A reader of this paper will notice that using the proposed approach assists
with the mean value of feature diffusion over use cases drop from 3.5 using the
PLUSS approach to 2 (on multimedia message product line) or from 1.3 to 1.2 (on
pedagogical product line). How significant are these results on these examples?
The benefits appear somewhat minor in these cases. Are there some other
difficulties using the proposed approach? Seems that this approach introduce
some hidden dependencies which might be difficult to follow.

Claiming that you are formally describing a modeling framework in section 3.2
may be a bit questionable. You just provide the implementation of weavers in
the functional language Haskell. Semantics of weavers is also not described
formally by functional code.

Running example 3.1 may not be that compelling. It can be shortened since
annotation [RegisterPreference] is included in Figure 7 and 8. Furthermore, the
description refers to some identifiers that are not explained. For example,
what does Step1M refer to in the statement "… and finishing at the Step 1M of
the Proceed to Purchase scenario …". The same is true for statement " …
ShipMethod is referenced in Step 4M of Figure 7". 

Comparing "from step and to step" from Figure 7 and right complete graphs on
Figure 12 the reader can notice that "to step" is not "END" as specified in
Figure 7, but R1 which was weaved afterward. Short comment is in order to
explain this (or to improve my understanding if I am wrong here).

Section 3.2 seems to be out of the scope of the paper. It just shows that you
test and verify your implementation in Haskell. The discussion in this section
has very minor impact to the main contribution of the paper. It also feels like
you just wanted brownie points for mentioning Haskell in the paper.

The related work section is a bit weak. There is no deep related work
comparison. Claims such as "we believe that the textual language is the
preferred representation for scenario specification" could be omitted since
they are not supported by experiments. Perhaps explain your view of textual
languages would be insightful to readers.


Minor problems:

I did not really understand the connection between Figures 1 and 2. Some of the
names are not the same and I was expecting some linkage.

Figure 11 seems to come out of the blue. This is some sort of configuration
tree control that appears to be a part of some tool that has yet to be
described.

Many figures (e.g., 1, 4, 10, 15) are too small and hard to read. Please
expand.

The order of features in Figure 10 (Shopping Cart, Bonus) and Figure 11 (Bonus,
Shopping Cart) is not the same.

Missing information about how constraints are described in a feature model
(Section 3.1.2).

I could not see what are 7 parameters (for each weaver) in the statement "We
represent each weaver by filling in the seven parameters of our 6-tuple
representation, …". Why is this number is explicitly mentioned?

Many references do not have page numbers (e.g., 2, 3, 4, 11, 12, 14, …)

          >>> Summary of the submission <<<

The authors appear to be saying the following. A product-line
definition has several components, including a feature model,
use-cases, etc. The addition of features impacts the use-cases in a
cross-cutting way. Therefore, it would be nice to have a language for
writing these better in the way we've grown accustomed to with
aspects.




          >>> Evaluation <<<

If my summary above is wrong, then I may have missed the point of this
paper. If it is right (or close enough to right), then I'm just not
convinced about the magnitude of the contribution here. The
specifications just aren't all that large; even your "industrial" spec
doesn't show a whole lot of size. Therefore, while this work would be
interesting at a much narrower conference, like SPLC or AOSD, but I
just don't see an OOPSLA paper's worth of contribution here.

Some small comments:

I don't see what's hard about the specification of the scenario
composer.

I would avoid calling the use of HUnit/QuickCheck "verification" --
surely you know better!

[9] is in ICFP, not ICSP. Citation [27] is completely scrambled. I
find it very strange that you list the publication venues by
abbreviation -- even for obscure ones -- but print the location, as if
one might fly to those cities to find the proceedings. If you want to
save space, please do so *usefully* (cut out the cities and give the
names of the conferences).

The English needs significant work.
